## 1. 발생한 문제 요약

| # | 문제 | 심각도 | 기능 영향 | 발견 방법 |
|---|------|--------|----------|----------|
| 1 | CSS 변수로 인한 디자인 토큰 검증 실패 | 낮음 | 없음 (정상 동작) | SSR HTML 검사 |
| 2 | 클라이언트 컴포넌트 SSR 검증 한계 | 중간 | 없음 (브라우저 정상) | curl 검사 |
| 3 | RSC payload 내 404 텍스트 오탐 | 중간 | 없음 (숨김 상태) | 사용자 보고 → Playwright 확인 |
| 4 | 디자인 사양 대소문자 불일치 | 낮음 | 없음 (미세 차이) | SSR HTML 검사 |
| 5 | verify-images.sh 코드 예시 URL 오탐 | 낮음 | 없음 | 스크립트 실행 |

**이전 세션 대비 변화**: 기능적 결함(이미지 깨짐, 404) → **검증 도구의 정밀도** 문제로 성격이 바뀜.

## 2. 이전 교훈 적용 효과 평가

### 적용된 교훈과 결과

| 교훈 | 적용 여부 | 효과 |
|------|----------|------|
| 브라우저 검증 의무화 | 적용 | 이미지 깨짐 문제 **0건** (이전: 2건) |
| 이미지 다운로드+내용 확인 | 적용 | 이미지 내용 불일치 **0건** (이전: 1건) |
| verify-images.sh 자동화 | 적용 | 11개 이미지 전체 HTTP 200 자동 확인 |
| Playwright 스크린샷 | 적용 | 404 오탐을 즉시 해소 (이전이면 수동 확인 불가) |

### 결론: **교훈 적용은 효과적**

이전 세션에서 발생한 **핵심 결함(이미지 깨짐, 내용 불일치)이 0건**으로 감소.
이번 문제들은 모두 **검증 도구의 정밀도** 문제이며, 기능적 결함이 아님.

## 3. 문제별 상세 분석 및 Claude-Gemini 합의

### 3.1 CSS 변수 vs hex 코드 검증 (Claude-Gemini 합의)

**문제**: 에이전트가 `var(--color-gold)` 로 구현했으나 검증 스크립트가 `#C4983B`를 검색

| | Claude | Gemini |
|---|--------|--------|
| CSS 변수 사용 | 더 좋은 관행 | 유지보수성·확장성에 바람직 |
| 해결 방향 | Playwright `getComputedStyle`로 검증 | 동일 (브라우저 기반 검증 전환) |

**합의된 해결책**: 스타일 검증은 SSR HTML grep 대신 **Playwright `getComputedStyle` 기반**으로 전환

### 3.2 SSR vs CSR 검증 전략 (Claude-Gemini 합의)

**합의: 하이브리드 전략**

```
[빠른 검증] curl 기반 SSR
  └─ SEO 메타태그, 초기 콘텐츠, 서버 렌더링 데이터
  └─ 속도 우선, CI 초기 단계

[정밀 검증] Playwright 기반 CSR/E2E
  └─ 클라이언트 렌더링 콘텐츠, CSS 스타일, 이미지 로드
  └─ 정확도 우선, QA 단계
```

**역할 분담 원칙**: curl은 "존재 여부", Playwright는 "동작 여부"

### 3.3 검증 스크립트 오탐 (Claude-Gemini 합의)

| 방법 | 설명 | 합의 |
|------|------|------|
| 스캔 대상 제한 | 데이터 파일(`src/data/`)만 스캔 | **1순위** |
| 무시 패턴 | `example.com`, `placeholder` 등 제외 | **2순위** |
| 정규식 개선 | `imageUrl:` 뒤의 URL만 추출 | **3순위** |

### 3.4 디자인 사양 충실도 (Gemini 추가 관점)

Gemini 권장:
- 정확한 텍스트는 **백틱으로 감싸서** 에이전트에게 전달: `` `CURATED LIFESTYLE` ``
- JSON/YAML 형식으로 텍스트 데이터를 직접 제공하여 에이전트가 복사-붙여넣기하도록 유도
- Evaluator가 **대소문자 구분(case-sensitive) 정확한 문자열 매칭** 수행

### 3.5 RSC payload 404 오탐 (Claude 독자 분석)

**원인**: Next.js가 SSR HTML의 `<script>` 태그 내 RSC 직렬화 데이터에 not-found fallback을 포함

**대응 방법**:
- SSR HTML에서 404 검색 시 `<script>` 태그 내부 제외
- 또는 Playwright에서 `textContent`에 404가 **실제로 보이는지** `display`/`visibility` 속성 확인
- 이번에 사용한 방법(DOM 가시성 분석)이 올바른 접근

| 우선순위 | 문제 | 심각도 | 조치 |

## 5. 세션 간 진화 비교

### 1차 (RecipeHub 최초 구현)

```
문제 유형: 핵심 기능 결함
- next/image 400 에러 (이미지 깨짐)
- QA 단계 생략 (파이프라인 미준수)
- 잡채 이미지 내용 불일치
발견 주체: 사용자 3건 / 자동 0건
```

### 2차 (SimpleShop 리뉴얼)

```
문제 유형: 검증 도구 정밀도
- CSS 변수 미인식 (검증 도구 한계)
- CSR 콘텐츠 SSR 검증 불가 (검증 범위 한계)
- RSC payload 404 오탐 (검증 정확도 한계)
- 스크립트 오탐 (검증 대상 범위 과다)
발견 주체: 자동 검증 3건 / 사용자 1건 (404 오탐 → 즉시 해소)
```

### 진화 방향

```
1차: "동작하지 않는 앱" → 사용자가 발견
2차: "동작하지만 검증이 부정확" → 대부분 자동 발견
```

**핵심 결함은 0건으로 감소. 문제의 성격이 "기능 결함"에서 "검증 정밀도"로 이동.**
