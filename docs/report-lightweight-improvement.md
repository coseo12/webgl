> 대상: 하네스 워크플로우 템플릿 v2.1

경량화(4,990줄→1,256줄, 에이전트 9→1, 스킬 15→5) 수행 후 두 가지 문제가 식별됨:

1. **코드 변경 시 관련 문서가 자동 업데이트되지 않는 문제** — docs/ 하위 5건에서 삭제된 에이전트 참조 잔존
2. **Anthropic 원칙 대비 A 미달 항목 3건** — 분리된 평가자(A-), 스프린트 계약(A-), 구체적 버그 추적(B+)

### Claude-Gemini 합의

**핵심 방안: CI 검증 스크립트**

두 모델 모두 CI에서 "삭제된 파일 참조 검출" 스크립트를 실행하는 것이 최적이라는 데 합의.

### Claude 분석

**Gemini의 제안은 대체로 타당하지만, 현재 프로젝트 규모에서는 과도한 부분이 있음.**

1. **CI 스크립트**: 합의. 다만 구현은 복잡한 파이프라인이 아니라 **단순 grep 스크립트**로 충분.
   ```bash
   # 삭제된 에이전트/스킬이 docs/에서 참조되는지 확인
   # 이 정도면 현재 규모에서 충분
   grep -rn "Orchestrator\|Evaluator\|QA 에이전트" docs/ && exit 1
   ```

2. **Pre-commit hook**: Gemini는 husky 도입을 제안했으나, 1인 개발에서 **CLAUDE.md에 규칙 추가**로 충분. Claude Code가 CLAUDE.md를 읽고 따르므로, "코드/설정 삭제 시 docs/ 하위 관련 문서를 확인하고 업데이트한다"는 규칙 자체가 hook 역할.

3. **코드→문서 자동 생성**: 과도한 엔지니어링. 10파일 1,256줄 규모에서 불필요.

4. **구조화된 참조 린터**: Gemini의 `[[agent:my_agent_id]]` 제안은 새 문법 도입 → 학습 비용 → 1인 개발에서 비효율적.

### 최종 결정

| 우선순위 | 방안 | 구현 방법 |
|---------|------|----------|
| 1 | CLAUDE.md 규칙 추가 | "코드/설정 삭제 시 docs/ 관련 문서 확인" 규칙 1줄 |
| 2 | CI 스크립트 (선택) | 삭제된 에이전트명 grep → 실패 시 경고 |
| 3 | 문서 최소화 유지 | 이미 수행됨, 추가 조치 불필요 |

## 주제 2: A 미달 항목 개선

### 2-1. 분리된 평가자 (A- → ?)

**현재 상태**: cross-validate가 평가자 역할, Gemini 429 실패 시 "Claude 단독 분석"으로 스킵

| 관점 | Gemini 제안 | Claude 분석 |
|------|-----------|-----------|
| 재시도 로직 | 지수 백오프 3~5회 | **이미 구현됨** (cross_validate.sh에 MAX_GEMINI_RETRIES=2) |
| 회로 차단기 | 연속 실패 시 일시 차단 | **과도** — 세션당 호출이 적어 의미 없음 |
| 로깅/알림 | 슬랙 알림 | **과도** — 1인 개발에서 터미널 로그로 충분 |

**Claude 독자 의견**: A- → A로 올리기 위한 진짜 문제는 "Gemini 실패 시 대안이 없다"가 아니라 **"cross-validate가 유일한 평가 메커니즘이다"**라는 점.

**현실적 개선**:
- cross-validate 실패 시 Claude가 **자가 평가 경고를 의식하면서 자체 리뷰**를 수행하도록 CLAUDE.md에 명시
- 이것은 이미 "자가 평가 경고" 규칙으로 부분 충족됨
- **결론: 현재 A-가 이 규모에서 현실적 최선. A로 올리려면 Gemini API 안정성 개선이 필요하나 통제 범위 밖.**

### 2-2. 스프린트 계약 (A- → A)

**현재 상태**: CLAUDE.md + developer.md + PR 템플릿에 규칙 있으나 자동 강제 없음

| 관점 | Gemini 제안 | Claude 분석 |
|------|-----------|-----------|
| PR 본문 파싱 CI | 필수 섹션 존재 + 비어있지 않음 확인 | **타당하지만 우선순위 낮음** |
| 형식 강제 vs 품질 검증 | 형식은 자동, 품질은 사람 리뷰 | **합의** |

**Claude 독자 의견**: CI로 PR 본문을 파싱하는 것은 가능하나, **Claude Code가 CLAUDE.md 규칙을 읽고 스프린트 계약을 자동 수행하므로** 이미 "소프트 강제"가 작동 중. 실전 검증(simple-shop #43)에서도 스프린트 계약이 정상 수행됨.

**현실적 개선**:
- developer.md 워크플로우에 이미 "스프린트 계약" 단계가 2번째로 포함됨
- **결론: Claude Code가 에이전트 정의를 따르므로 사실상 자동 강제됨. A- → A로 판정 가능.**

### 2-3. 구체적 버그 추적 (B+ → A)

**현재 상태**: "원인+수정점 명시" 규칙 있으나 적용 여부 검증 없음

| 관점 | Gemini 제안 | Claude 분석 |
|------|-----------|-----------|
| 버그 수정 전용 PR 템플릿 | Root Cause + Proposed Fix 섹션 | **과도** — 범용 템플릿으로 충분 |
| CI 검증 (fix: PR 대상) | 필수 섹션 존재 확인 | **실용적이나 1인 개발에서 비용 대비 가치 낮음** |

**Claude 독자 의견**: B+인 진짜 원인은 "규칙은 있으나 실천의 증거가 부족"한 것. 실전 검증(simple-shop #43)에서는 스프린트 계약이 작동했으므로, **동일 메커니즘이 버그 추적에도 적용 가능**.

**현실적 개선**:
- CLAUDE.md의 스프린트 계약 규칙 3번에 이미 "기준 미충족 시 구체적 피드백과 함께 반려 — 단순 실패가 아닌 원인+수정점 명시" 포함
- developer.md에 **fix 커밋 시 원인 분석 포함** 규칙 1줄 추가로 충분
- **결론: B+ → A 달성 가능. 규칙 명시 + 실전 검증 조합.**

## 핵심 인사이트

> **Claude Code 환경에서 "자동 강제"의 정의가 다르다.**
> 전통적 소프트웨어에서 자동 강제 = CI/hook. 하지만 Claude Code에서는
> **CLAUDE.md + 에이전트 정의를 AI가 읽고 따르는 것 자체가 소프트 자동 강제**다.
> 1인 개발자-AI 페어 프로그래밍에서 CI 파이프라인을 구축하는 것보다
> CLAUDE.md에 규칙 1줄을 추가하는 것이 더 효과적이고 비용 효율적이다.
