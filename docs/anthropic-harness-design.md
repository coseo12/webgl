# Anthropic: Harness Design for Long-Running Application Development

> 원문: https://www.anthropic.com/engineering/harness-design-long-running-apps
> 발행일: 2026-03-24 | 저자: Prithvi Rajasekaran (Anthropic Labs)
> 이 문서는 원문 접근 불가 시를 대비한 핵심 내용 아카이브입니다.

---

## 개요

두 가지 상호연관된 문제를 해결:
1. Claude가 고품질 프론트엔드 디자인을 생성하도록 함
2. 인간의 개입 없이 완전한 애플리케이션을 구축하도록 함

GAN(생성 적대 신경망)에서 영감을 받아, **생성기(Generator)와 평가기(Evaluator)를 분리한 다중 에이전트 구조**를 설계.

---

## 순진한 구현이 실패하는 이유

### 문제 1: 컨텍스트 창 채움으로 인한 일관성 손실

> "모델들은 컨텍스트 창이 채워지면서 긴 작업에서 일관성을 잃는 경향이 있으며,
> 일부 모델은 컨텍스트 한계에 접근할 때 조기에 작업을 종료하는 '컨텍스트 불안'을 보인다"

**해결책**: 컨텍스트 리셋 — 컨텍스트 창을 완전히 비우고 구조화된 아티팩트로 상태를 이월.
압축(compaction)은 연속성을 유지하지만 컨텍스트 불안감이 지속되므로, 장시간 작업에는 리셋이 더 효과적.

### 문제 2: 자가평가 편향

> "자신이 생산한 작업을 평가하도록 요청받으면, 에이전트들은 인간 관찰자에게는
> 명백히 평범해 보이는 경우에도 작업을 자신감 있게 칭찬한다"

**해결책**: 평가 역할을 분리된 에이전트에게 위임. 분리된 평가기가 생성기보다 더 비판적 판단 가능.

---

## 프론트엔드 디자인: 주관적 품질을 채점 가능하게 만들기

### 4가지 채점 기준 (루브릭)

| 기준 | 설명 |
|------|------|
| **Design Quality** | 색상, 타이포그래피, 레이아웃, 이미지가 결합되어 특정 분위기와 정체성을 만드는가 |
| **Originality** | 맞춤형 결정의 증거가 있는가, 아니면 템플릿/라이브러리 기본값/AI 생성 패턴인가 |
| **Craft** | 타이포그래피 계층, 간격 일관성, 색상 조화, 대비 비율의 기술적 실행 |
| **Functionality** | 사용자가 인터페이스를 이해하고, 주요 작업을 찾으며, 추측 없이 완료할 수 있는가 |

### 실행 방식

- Claude Agent SDK를 사용하여 생성기-평가기 루프 구축
- 평가기는 **Playwright MCP**로 라이브 페이지와 상호작용
- 5~15회 반복 실행
- 전체 실행 시간: 최대 4시간

### 주목할 만한 결과

네덜란드 미술관 웹사이트 — 9차 반복까지 깔끔한 다크테마 랜딩페이지였지만,
10차 반복에서 **접근 방식 자체를 폐기하고 공간 경험으로 재구상**:
CSS 원근감으로 렌더링된 체크무늬 바닥의 3D 방, 벽에 걸린 미술작품, 입구 기반 네비게이션.

## 풀스택 코딩으로의 확장

### 아키텍처: 3개 에이전트

**1. 플래너 에이전트**
- 1~4줄 프롬프트를 완전한 제품 사양으로 확장
- 기술 구현보다 **제품 맥락 및 높은 수준의 기술 설계**에 집중
- AI 기능을 제품 사양에 직조하는 기회 발굴
- **고수준 스펙 유지**: 상세한 기술 명세는 오류 전파 위험을 증가시키므로 배제

**2. 생성기 에이전트**
- React, Vite, FastAPI, SQLite/PostgreSQL 스택
- **스프린트별로 한 번에 한 기능씩** 작업
- 각 스프린트 후 자가평가 실행
- Git 버전 관리 사용

**3. 평가기 에이전트**
- Playwright MCP로 실행 중인 애플리케이션 클릭하여 테스트
- UI 기능, API 엔드포인트, 데이터베이스 상태 모두 검증
- **스프린트 계약(Sprint Contract)**: 코드 작성 전에 완료 조건을 정의하고 합의

---

## 스프린트 계약 (Sprint Contract)

코딩 전에 생성기와 평가기가 "done"의 정의에 합의.

- 각 항목에 **검증 가능한 기준** 포함
- 통과하지 못한 기준은 **구체적 피드백과 함께 반려**
- 표면적 테스트가 아닌 **엣지 케이스까지 탐색**

### 발견된 버그 예시 (레트로 게임 메이커)

| 계약 기준 | 평가기 발견 |
|----------|-----------|
| 직사각형 채우기 도구로 드래그하여 영역 채우기 | **실패** — 도구가 드래그 시작/끝점에만 타일 배치 |
| 엔티티 스폰 포인트 선택 및 삭제 | **실패** — 삭제 키 핸들러가 `selection`과 `selectedEntityId` 모두 필요 |
| API를 통해 애니메이션 프레임 재정렬 | **실패** — FastAPI가 'reorder'를 frame_id 정수로 매칭, 422 반환 |

## 평가기 성능 조정

### 초기 문제

> "초기 실행에서 평가기가 정당한 문제를 파악했지만, 이를 큰 문제가 아니라고
> 생각하고 작업을 승인했다"

### 해결 과정

1. 평가기의 로그 검토
2. 평가자의 판단이 자신의 판단과 다른 예시 찾기
3. QA 프롬프트 업데이트 (few-shot 예제로 기준 정렬)
4. 여러 발전 루프를 거쳐 합리적인 채점에 도달
5. **반복적 프롬프트 개선**으로 정확도 향상 (즉시 완벽하지 않음)

## 단일 에이전트 vs 전체 하네스 비교 (레트로 게임 메이커)

| 하네스 | 실행 시간 | 비용 |
|--------|---------|------|
| 단일 에이전트 | 20분 | $9 |
| 전체 하네스 | 6시간 | $200 |

**단일 에이전트의 문제점:**
- 레이아웃이 공간을 낭비
- 워크플로우가 경직
- **핵심 기능 손상**: 엔티티가 화면에 나타나지만 입력에 응답하지 않음 (display-only)
- 엔티티 정의와 게임 런타임 간의 배선 끊김

**전체 하네스의 장점:**
- 폴리시되고 부드러운 인터페이스
- 일관된 시각적 정체성
- 더 풍부한 스프라이트 에디터
- Claude 통합을 통한 AI 기능
- **실제 작동하는 플레이 모드**

|----------------|---------|------|
| 플래너 | 4.7분 | $0.46 |
| 빌드 (1차) | 2시간 7분 | $71.08 |
| QA (1차) | 8.8분 | $3.24 |
| 빌드 (2차) | 1시간 2분 | $36.89 |
| QA (2차) | 6.8분 | $3.09 |
| 빌드 (3차) | 10.9분 | $5.88 |
| QA (3차) | 9.6분 | $4.06 |
| **전체** | **3시간 50분** | **$124.70** |

### 평가기 피드백 (1차)

> "강력한 앱으로 우수한 디자인 충실도, 견고한 AI 에이전트, 양호한 백엔드를 갖추고 있습니다.
> 주요 실패 지점은 기능 완성도입니다 — 앱이 인상적이고 AI 통합이 잘 작동하지만,
> 여러 핵심 DAW 기능이 깊이 있는 상호작용 없이 **디스플레이 전용**입니다"

### 평가기 피드백 (2차) — 남은 문제

- 오디오 녹음이 여전히 스텁 전용
- 클립 리사이즈 및 클립 분할 미구현
- 효과 시각화가 수치 슬라이더만, 그래픽 없음

## 하네스 진화: Opus 4.6으로의 전환

### Opus 4.6 개선 사항

- 더 신중한 계획
- 에이전트 작업의 더 오랜 지속
- 더 큰 코드베이스에서 더 신뢰할 수 있는 작동
- 자신의 실수를 잡기 위한 더 나은 코드 검토 및 디버깅
- 긴 컨텍스트 검색 상당히 개선

### 결과: 스프린트 구조 제거 가능

- Opus 4.6에서는 컨텍스트 리셋 불필요 (자동 컴팩션으로 충분)
- 스프린트 분해 없이도 일관된 결과
- 평가기 활용도는 작업 난도에 따라 가변적

## 핵심 원칙

### 1. 최소 복잡성 원칙

> "find the simplest solution possible, and only increase complexity when needed"
> — 가장 단순한 솔루션을 찾고, 필요할 때만 복잡성을 증가시킨다

### 2. 하네스 구성요소 = 모델 한계의 가정

> "every component in a harness encodes an assumption about what the model can't do"
> — 하네스의 모든 구성요소는 모델이 혼자 할 수 없는 것에 대한 가정을 인코딩한다

새 모델 출시 시 각 구성요소의 필요성을 재평가해야 한다.

### 3. 분리된 평가자의 필수성

자기 평가는 과도하게 긍정적. 별도 평가자를 통한 객관적 검증이 필수.
평가기 미세조정이 생성기 자체 비판보다 훨씬 실질적.

### 4. 스프린트 계약의 가치

코딩 전 "done"을 정의. 각 기준은 검증 가능해야 하며,
실패 시 구체적 피드백(원인 + 수정점)을 제공.

### 5. 표면적 테스트 회피

> "tended to test superficially, rather than probing edge cases"
> — 표면적 테스트가 아닌 엣지 케이스 탐색이 필요

### 6. display-only 버그 패턴 인식

UI는 완성되었으나 실제 기능은 스텁 상태로 남는 패턴이 반복됨.
버튼이 렌더링되지만 이벤트 핸들러 없음, 폼이 보이지만 제출 미동작 등.

### 7. 구체적 버그 추적

단순 "버그 발견"이 아닌, 원인과 수정점을 명시하여
생성기가 추가 조사 없이 바로 수정 가능한 수준으로 보고.

### 8. 점진적 개선

완전한 QA를 한 번에 수행하기보다 **여러 라운드 순환**.
초기 라운드에서 기능 완성도, 후속 라운드에서 엣지 케이스 검증.

## 향후 전망

> "모델이 계속 개선되면서, 더 오래 일하고 더 복잡한 작업을 기대할 수 있다.
> 경우에 따라 모델 주변의 스캐폴드가 시간이 지남에 따라 덜 중요해지는 것을 의미한다.
> 반면, 모델이 더 나아질수록 모델이 기준선에서 혼자 할 수 없는 복잡한 작업을
> 달성할 수 있는 하네스를 개발할 더 많은 공간이 있다."
