# 모델 가정 문서

> 각 구성요소가 모델의 어떤 한계를 보완하는지 명시한다.
> 새 모델 출시 시 이 문서를 기준으로 구성요소의 필요성을 재평가한다.
>
> 참조: https://www.anthropic.com/engineering/harness-design-long-running-apps

## 기준 모델

- **현재 기준**: Claude Opus 4.6 (2026-03 기준)
- **마지막 재평가**: 2026-03-28

## 구성요소별 모델 가정

| 구성요소 | 보완하는 모델 한계 | 재평가 시점 |
|---------|-----------------|-----------|
| Developer 에이전트 | 워크플로우 규칙(브랜치, 커밋, 검증 단계)을 일관되게 따르는 능력 | 모델이 규칙 없이도 일관된 워크플로우를 유지할 때 |
| 스프린트 계약 | 구현 전 완료 기준을 자발적으로 정의하는 능력. "done"의 정의 없이 표면적 구현에 그치는 경향 | 모델이 자발적으로 검증 가능한 완료 기준을 설정할 때 |
| 3단계 브라우저 검증 | "빌드 성공 = 동작함"이라는 잘못된 가정. display-only 버그를 인지하지 못하는 경향 | 모델이 브라우저 검증 없이도 기능 동작을 보장할 때 |
| 디자인 품질 루브릭 | AI 생성 코드의 디자인 품질이 템플릿/기본값에 의존하는 경향. 독창성 부족 | 모델이 독창적이고 일관된 디자인을 자발적으로 생성할 때 |
| cross-validate | 자기 평가 편향 (self-evaluation bias). 자신의 작업을 과도하게 긍정 평가하는 현상 | 자기 평가 정확도가 외부 평가와 동등해질 때 |
| 자가 평가 경고 | 기본적으로 관대하게 평가하는 경향. 문제를 발견해도 "괜찮다"고 자기설득하는 현상 | 모델이 기본적으로 엄격한 평가 태도를 보일 때 |
| CLAUDE.md 실전 교훈 | HTTP 200 ≠ 올바른 리소스, 렌더링 ≠ 동작 등 반복되는 실패 패턴을 자발적으로 인지하지 못하는 한계 | 모델이 이러한 패턴을 사전 지식으로 내재화할 때 |

## 재평가 프로토콜

1. 새 모델 출시 시 이 문서를 검토한다
2. 각 구성요소에 대해 **제거 실험**을 수행한다:
   - 구성요소를 제거한 상태에서 동일 태스크를 실행
   - 결과 품질을 비교
   - 품질 저하가 없으면 해당 구성요소 제거 검토
3. 한 번에 하나의 구성요소만 제거하여 개별 영향을 측정한다

## 핵심 원칙

> "every component in a harness encodes an assumption about what the model can't do"
> — Anthropic, Harness Design for Long-Running Apps

불필요한 구성요소는 과감히 제거한다. 최소 복잡성이 목표다.
