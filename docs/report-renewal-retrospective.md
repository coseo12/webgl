## 2. 근본 원인 분석

### 2.1 "빌드+테스트 통과 = 동작" 자기설득

| 검증 수단 | 실제 커버리지 | 놓치는 것 |
|----------|------------|----------|
| `npm run build` | 타입, 라우트 정적 생성 | 런타임 이미지 로드, CSS 렌더링 |
| `vitest` (jsdom) | DOM 구조, 로직 | 실제 이미지 로드, 네트워크 요청, CSS 레이아웃 |
| HTTP 상태 코드 | URL 접근 가능 여부 | 이미지 내용의 적합성 |
| SSR HTML 파싱 | 서버 렌더링 내용 | 클라이언트 컴포넌트 렌더링 결과 |

**핵심**: 모든 자동 검증을 통과해도 "사용자가 보는 화면"과의 격차가 존재한다.

### 2.2 검증 단계별 격차 지도

```
Developer 구현 완료
  ↓
[1] 빌드 (npm run build)          ← 타입/구조만 검증
  ↓
[2] 단위 테스트 (vitest)           ← 로직만 검증, DOM 구조만 확인
  ↓
[3] HTTP 검증 (curl)              ← URL 접근만 확인
  ↓
[GAP] ────────────────────────────── 여기서부터 자동 검증 불가 ──
  ↓
[4] 이미지 내용 일치              ← 사람 눈으로만 확인 가능
[5] CSS 레이아웃 렌더링           ← 브라우저 필요
[6] 반응형 뷰포트별 레이아웃       ← 브라우저 + 뷰포트 전환 필요
[7] 디자인 품질 (미학적 판단)      ← 사람 판단 필요
```

## 3. Claude-Gemini 합의 사항

### 3.1 이미지 선택 프로세스 개선 (합의)

| 항목 | Claude | Gemini | 합의 |
|------|--------|--------|------|
| 이미지 다운로드 후 직접 확인 | 후보 4개 다운 → 눈으로 선택 | AI 비전 모델로 내용 분석 | **다운로드+확인 필수** |
| HTTP 200만으로 불충분 | 동의 | 강력 동의 | **합의** |
| 메타데이터 활용 | 미언급 | 태그/설명 교차 검증 제안 | Gemini 추가 관점 |

**현재 적용 가능한 프로세스**:
```
키워드 검색 → 후보 여러 개 다운로드 → 이미지 직접 확인(Read) → 적합한 것 선택
```

### 3.2 브라우저 검증 자동화 (합의)

| 방법 | 현재 가능 | 추가 구현 필요 |
|------|----------|--------------|
| HTTP 상태 코드 | 가능 | - |
| SSR HTML 파싱 | 가능 | - |
| API 응답 검증 | 가능 | - |
| 헤드리스 브라우저 스크린샷 | **가능** (Playwright) | QA 에이전트에 통합 |
| 시각적 회귀 테스트 | 미구현 | 기준 이미지 비교 도구 필요 |
| AI 비전 분석 | 부분 가능 (Read로 이미지 확인) | 자동 판정 로직 필요 |

**Gemini 핵심 제안**: "비주얼 QA 게이트" — UI 커밋 전 헤드리스 브라우저 기반 시각적 검증을 필수 게이트로 설정

### 3.3 파이프라인 준수 강제 (합의)

| 항목 | Claude | Gemini |
|------|--------|--------|
| 문제 | 속도 위해 QA 생략 | 에이전트의 워크플로우 임의 변경 |
| 조치 | Developer 프롬프트에 브라우저 검증 의무화 | 준수 모니터링 + 강제 중단 시스템 |
| 한계 | 프롬프트 지시는 "권고"에 가까움 | 강제 메커니즘 구현 필요 |

**Claude 추가 분석**: `dispatch-agent.sh`로 에이전트를 실행하면 파이프라인이 강제되지만, 수동 작업(에이전트 병렬 실행) 시 단계 생략이 가능한 구조적 약점이 존재.

## 6. 프레임워크 간소화 종합 평가

### 검증된 강점

1. **적응형 파이프라인 동작 확인**: size:s(4단계)~size:l(6단계)이 예제에서 자연스럽게 분배됨
2. **Evaluator 통합 효과**: Auditor+Reviewer 분리 시 발생하던 중복 검증이 해소
3. **피드백 루프 작동**: 문제 발견 → 원인 분석 → 프롬프트 개선 → 재검증 사이클이 실제 동작
4. **Spec-Driven Testing 유효**: 시나리오 9~10개 → 테스트 13~20개 → 구현 순서가 효과적

### 남은 약점

1. **시각적 검증 격차**: 자동 검증(빌드+테스트)과 실제 화면 사이의 GAP
2. **외부 에셋 내용 검증 부재**: HTTP 200 ≠ 올바른 이미지
3. **파이프라인 준수 강제력 부족**: 수동 작업 시 단계 생략 가능
4. **디자인 품질의 주관적 평가**: CLI에서 미학적 판단 불가

### 이번 세션의 교훈

> **"테스트는 버그가 없음을 증명하지 못한다. 테스트가 찾도록 설계된 버그만 찾는다."**
> — Dijkstra

20개 테스트가 통과해도 이미지가 깨졌고, HTTP 200이어도 잡채 사진이 아니었다.
프레임워크의 다음 과제는 **"검증이 커버하지 못하는 영역을 인식하고, 그 영역에 대해 명시적으로 확인하는 프로세스"** 를 만드는 것이다.
