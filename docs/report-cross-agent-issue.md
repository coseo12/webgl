# 교차검증 보고서: 크로스-에이전트 이슈 자동 생성 메커니즘

> 검증 일시: 2026-03-27
> 검증 유형: architecture (아키텍처 설계 검증)
> 검증 모델: Claude Opus 4.6 + Gemini 2.5 Flash

---

## 1. 제안 내용

에이전트가 작업 중 **다른 영역의 에이전트가 처리해야 할 문제를 발견**했을 때, GitHub 이슈를 자동 생성하고 해당 에이전트에게 넘기는 구조.

### 시나리오 예시

| 발견자 | 발견 내용 | 대상 에이전트 |
|--------|----------|-------------|
| Frontend Dev | API 스펙 불일치 | Backend Dev |
| QA | 설계 누락 | Architect |
| Auditor | 보안 취약점 | DevOps |
| Reviewer | 문서 불일치 | Integrator |
| Backend Dev | DB 스키마 변경 필요 | Architect |

### 현재 처리 방식

| 메커니즘 | 사용 위치 | 한계 |
|----------|----------|------|
| PR 코멘트 | QA 결과 보고, Reviewer 피드백 | PR 닫히면 추적 어려움 |
| 오케스트레이터 보고 | Developer의 충돌 감지 | 병목, 컨텍스트 손실 |
| status:blocked 라벨 | DevOps의 장애 처리 | 이슈 간 연결 없음 |

## 2. 효율성 분석

### Claude 분석

**조건부 긍정.** 하지만 현재 프레임워크 규모에서는 오버엔지니어링 위험이 있다.

**효율적인 경우:**
- 병렬 개발 시 FE/BE 간 API 스펙 불일치 (즉시 공유해야 함)
- QA가 설계 수준 결함을 발견 (파이프라인 역방향 전달 필요)
- 보안 취약점 발견 (긴급도 높음, 즉시 전파)

**비효율적인 경우:**
- 단순 코드 스타일 이슈 (PR 코멘트로 충분)
- 다음 파이프라인 단계에서 자연스럽게 발견될 문제
- 에이전트 판단 오류로 인한 잘못된 이슈 생성

### Gemini 분석

**장기적으로 효율 증대.** 단, "의미 있는" 이슈만 생성되도록 제어해야 함.
- 조기 발견으로 후반 단계 비용 절감
- 병목 현상 완화 (순차 파이프라인의 한계 보완)
- 책임 분할의 명확화

### 합의

**무조건 모든 문제를 이슈로 만드는 것은 비효율적.** 심각도 기반 필터링 필수.

## 3. 핵심 트레이드오프 분석

### 3.1 이슈 폭증 위험

| 접근법 | 장점 | 단점 |
|--------|------|------|
| 모든 발견을 이슈화 | 누락 없음 | 노이즈 폭증, 관리 불가 |
| 심각도 필터링 | 중요 이슈만 추적 | 경미한 문제 누락 가능 |
| Orchestrator 승인 | 품질 보장 | 병목, 지연 |

**Claude 권장:** 3단계 분류

```
Critical/Blocking → 이슈 생성 + 파이프라인 중단
Major             → 이슈 생성 + 작업 계속
Minor             → PR 코멘트로 처리 (이슈 생성 안 함)
```

**Gemini 동의:** 중복 이슈 검사 + 생성 빈도 제한도 추가 필요

### 3.2 파이프라인 중단 vs 계속

| 심각도 | 행동 | 예시 |
|--------|------|------|
| Critical (Blocking) | 중단 + 이슈 생성 + Orchestrator 알림 | API 엔드포인트 미존재 |
| Major (Non-blocking) | 이슈 생성 + 작업 계속 | 에러 처리 누락 |
| Minor | PR 코멘트 + 작업 계속 | 변수명 컨벤션 |

**Claude & Gemini 합의:** 심각도 + 종속성으로 판단. 일률적 규칙 불가.

### 3.3 순환 참조 위험

```
Frontend Dev → "API 스펙 틀림" → Backend Dev
Backend Dev → "프론트 요청이 잘못됨" → Frontend Dev
→ 무한 루프
```

**방지책 (합의):**
1. 이슈에 `parent_issue` 필드로 연결 — 같은 체인에서 재할당 제한
2. 동일 에이전트 쌍 간 N회(3회) 반복 시 Orchestrator 강제 개입
3. 이슈 생성 시 "해결 목표(Definition of Done)" 필수

### 3.4 Orchestrator 역할

| 방식 | 장점 | 단점 |
|------|------|------|
| 에이전트 직접 통신 | 빠름, 병목 없음 | 통제 어려움, 순환 위험 |
| Orchestrator 중재 | 안정적, 일관성 | 병목, 지연 |
| **하이브리드** | 속도 + 안정성 | 구현 복잡도 |

**Claude & Gemini 합의: 하이브리드 접근**

```
Major/Minor 이슈:
  에이전트가 직접 이슈 생성 → Orchestrator에 비동기 알림

Critical/Blocking 이슈:
  에이전트가 Orchestrator에 보고 → Orchestrator가 이슈 생성 + 할당 + 파이프라인 조정
```

## 4. 대안 비교

| 방식 | 추적성 | 속도 | 관리 비용 | 적합한 경우 |
|------|--------|------|-----------|------------|
| **PR 코멘트** | 낮음 | 빠름 | 없음 | 코드 수준 피드백, Minor |
| **Orchestrator에만 보고** | 중간 | 느림 | 낮음 | Critical, 파이프라인 조정 필요 |
| **크로스 이슈 자동 생성** | 높음 | 중간 | 중간 | Major, 독립 추적 필요 |
| **하이브리드 (3가지 혼용)** | 높음 | 빠름 | 높음 | 모든 경우 |

### Claude 고유 분석: 현재 프레임워크에서의 현실성

현재 에이전트들의 실제 코드를 확인한 결과:
- `frontend-developer.md`, `backend-developer.md`: 충돌 시 **오케스트레이터에 보고**만 함
- `qa.md`: 발견 이슈를 **PR 코멘트로 보고**만 함
- `devops.md`: `status:blocked` 라벨만 사용
- **크로스 이슈 생성 로직은 어떤 에이전트에도 없음**

즉, 현재는 "문제를 발견해도 자기 범위에서만 처리하거나 오케스트레이터에 보고"하는 구조. 다른 에이전트에게 직접 작업을 위임하는 메커니즘이 부재.

## 5. 최종 권장안

### 결론: **제한적 도입 권장 (하이브리드 3단계)**

전면 도입은 오버엔지니어링. 심각도 기반 3단계로 나눠서 도입.

### 구체적 설계

```
에이전트가 다른 영역의 문제 발견
  │
  ├─ Minor (코드 스타일, 경미한 개선)
  │   → PR 코멘트로 처리 (현재와 동일)
  │   → 이슈 생성 안 함
  │
  ├─ Major (기능 누락, 스펙 불일치, 에러 처리 부재)
  │   → GitHub 이슈 자동 생성
  │   → 라벨: type:cross-issue, agent:<대상>, priority:<레벨>
  │   → 본문: 발견 컨텍스트, 관련 파일, 재현 방법
  │   → 자기 작업은 계속 진행
  │   → Orchestrator에 비동기 알림 (state.json 업데이트)
  │
  └─ Critical (블로킹, 보안, 데이터 무결성)
      → Orchestrator에 즉시 보고
      → Orchestrator가 이슈 생성 + 에이전트 할당 + 파이프라인 조정
      → 자기 작업 중단, 해결 대기
```

### 필요한 변경 사항

| 변경 대상 | 내용 |
|-----------|------|
| CLAUDE.md | `type:cross-issue` 라벨 추가, 크로스 이슈 규칙 추가 |
| 각 에이전트 .md | "크로스 이슈 생성 규칙" 섹션 추가 |
| orchestrator.md | 크로스 이슈 모니터링 + 순환 참조 감지 로직 |
| create-issue 스킬 | cross-issue 타입 지원, parent_issue 연결 |
| setup-labels.sh | `type:cross-issue` 라벨 추가 |
| sync-status 스킬 | 크로스 이슈 상태 추적 |

### 순환 참조 방지 규칙

```
1. 크로스 이슈에는 parent_issue 링크 필수
2. 같은 parent에서 파생된 이슈 → 같은 에이전트에 재할당 금지
3. 동일 에이전트 쌍 간 3회 이상 왕복 → Orchestrator 강제 개입
4. 크로스 이슈의 크로스 이슈(2단계 이상) 생성 금지
```

## 6. Claude vs Gemini 비교

### 합의 사항

| 항목 | 합의 |
|------|------|
| 효율성 | 심각도 필터링 하에 긍정적 |
| 이슈 폭증 제어 | 임계값 + 중복 검사 + 빈도 제한 |
| 파이프라인 중단 | 심각도 기반 동적 판단 |
| Orchestrator 역할 | 하이브리드 (Major=직접, Critical=중재) |
| 순환 참조 | 재할당 제한 + 모니터링 |

### 이견

| 항목 | Claude | Gemini |
|------|--------|--------|
| 도입 범위 | 제한적 (3단계 분류) | 포괄적 (모든 이슈에 적용 후 필터링) |
| 우선순위 판단 | 규칙 기반으로 충분 | ML 기반 학습도 장기적으로 고려 |
| 현실성 | 현재 규모에서 Major만 우선 도입 | 전체 설계를 먼저 하고 점진적 도입 |

**Claude 추가 의견:** 현재 15개 에이전트, 단일 프로젝트 규모에서는 Orchestrator 중심 보고 방식으로도 충분히 동작함. 크로스 이슈는 **병렬 개발(FE+BE) 시나리오에서만 즉각적 가치가 있음.** 나머지는 파이프라인 순서상 다음 에이전트가 자연스럽게 감지.

## 7. 판정

| 판정 | 근거 |
|------|------|
| **조건부 도입 권장** | 효율성은 있으나 현재 규모에서 전면 도입은 오버엔지니어링. Major 이슈에 한해 우선 도입하고, 사용 패턴을 관찰한 후 확대 결정 |

**우선 도입 대상:**
1. FE/BE 병렬 개발 간 API 스펙 불일치 (가장 빈번한 시나리오)
2. QA/Auditor가 설계 수준 결함 발견 시 Architect로 역방향 전달
3. 보안 취약점의 DevOps 즉시 전파

**보류:**
- Minor 이슈의 크로스 전달 (PR 코멘트로 충분)
- ML 기반 우선순위 판단 (규모가 커진 후)
- 2단계 이상 크로스 이슈 체인 (복잡도 과다)
